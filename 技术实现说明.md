# 技术实现说明

重点回答两个问题：

- 实时 2D 轨迹平滑是如何实现的
- `baseline` 在本项目里是如何定义的

## 1. 总体架构

当前仓库有三条线：

- `src/drone_kalman_filter`
  - 实时主链路
- `tooling/drone_kalman_validation`
  - 离线验证与技术诊断
- `tooling/drone_kalman_visualization`
  - 面向前端观感的轨迹对比图

当前真正决定前端轨迹效果的是 `src/drone_kalman_filter`。

## 2. 实时主链路

主入口在：

- [plugin.py](D:/Github/Drone-Kalman-Filter/src/drone_kalman_filter/plugin.py)

整体流程：

```text
单条消息
  -> parse_message
  -> 按 (targetId, deviceId) 找状态
  -> 按 traceId / 时间 gap 判断是否切段
  -> 进入当前段的 smoother
  -> 生成平滑后位置
  -> 按 arrival_seq 保序释放
```

当前有两种 smoother：

- `kalman`
- `robust_prefilter_kalman`

## 3. 状态隔离与切段

### 3.1 状态隔离键

实时状态按下面的键隔离：

```text
track_key = (targetId, deviceId)
```

这样做的前提是：

- 前端按 `deviceId` 分别画线
- 不做跨设备融合

### 3.2 切段规则

即使 `track_key` 不变，也不一定属于同一段轨迹。

以下任一条件成立，就会切新段：

```text
1. traceId 变化
2. eventTime 倒退或不递增
3. 当前点与上一点的时间差 > max_segment_gap_seconds
```

当前默认：

```text
max_segment_gap_seconds = 10.0
```

切段的原因是：

- 避免把断轨前后的观测强行接成一条运动过程
- 避免长间隔直接污染状态估计

## 4. 局部平面坐标

实现文件：

- [geo.py](D:/Github/Drone-Kalman-Filter/src/drone_kalman_filter/geo.py)

代码不会直接在经纬度上做滤波，而是先把当前段内的经纬度映射到局部平面坐标：

- `East (m)`
- `North (m)`

使用的是“小范围局部切平面”近似。写成工程上更好读的形式是：

```text
e = R · cos(lat0) · (lon - lon0)
n = R · (lat - lat0)
```

其中：

- `e`：相对段起点的东向位移，单位米
- `n`：相对段起点的北向位移，单位米
- `R`：地球半径
- `lat0, lon0`：这一段第一个点的纬度、经度
- `lat, lon`：当前点的纬度、经度

可以把它理解成：

- 经度差先乘 `cos(lat0)`，把经线收缩补偿掉
- 纬度差直接按地球半径换成米
- 这样就把原始 `lat/lon` 变成了更适合滤波的二维平面坐标

这样做的好处是：

- 坐标单位统一成米
- 速度、过程噪声、残差阈值更容易统一建模
- 数值稳定性比直接在经纬度上滤波更好

## 5. `kalman` 模式

实现文件：

- [segment.py](D:/Github/Drone-Kalman-Filter/src/drone_kalman_filter/segment.py)
- [kalman.py](D:/Github/Drone-Kalman-Filter/src/drone_kalman_filter/kalman.py)

### 5.1 状态模型

当前使用二维恒速模型。状态向量写成列向量更容易理解：

```text
      | e_k  |
x_k = | n_k  |
      | ve_k |
      | vn_k |
```

含义是：

- `e_k`：第 `k` 个时刻的东向位置
- `n_k`：第 `k` 个时刻的北向位置
- `ve_k`：第 `k` 个时刻的东向速度
- `vn_k`：第 `k` 个时刻的北向速度

这个模型背后的假设很简单：

- 位置会随速度推进
- 速度在短时间内近似保持不变
- 真正的不规则机动，由过程噪声去吸收

### 5.2 观测模型

当前只把位置当成观测量，不直接把速度当观测量：

```text
      | e_meas,k |
z_k = | n_meas,k |
```

观测矩阵就是“只取状态里的前两维”：

```text
    | 1  0  0  0 |
H = | 0  1  0  0 |
```

直观理解就是：

- 观测能告诉滤波器“你现在大概在哪”
- 速度不是直接量到的，而是通过状态演化慢慢估出来

### 5.3 状态转移矩阵

对时间步长 `dt`，状态转移矩阵为：

```text
    | 1  0  dt  0  |
    | 0  1  0   dt |
F = | 0  0  1   0  |
    | 0  0  0   1  |
```

读法是：

- 新位置 = 旧位置 + 速度 × dt
- 新速度 = 旧速度

代码对应：

- `_transition_matrix(dt)`

### 5.4 时间步长

代码里使用：

```text
dt_eff = max(eventTime[k] - eventTime[k-1], min_dt_seconds)
```

当前默认：

```text
min_dt_seconds = 0.3
```

这么做是为了避免极短时间间隔把隐含速度和残差放大得过头。

### 5.5 过程噪声

过程噪声用的是“白噪声加速度”模型。为了可读性，可以先看一维版本：

```text
Q_1d(dt) =
| dt^4 / 4   dt^3 / 2 |
| dt^3 / 2   dt^2     |
```

二维版本就是把这个一维矩阵分别放到东西、南北两个方向上：

```text
Q(dt) = sigma_a^2 · block_diag(Q_1d(dt), Q_1d(dt))
```

也就是：

```text
Q(dt) = sigma_a^2 ·
| dt^4/4   0        dt^3/2   0      |
| 0        dt^4/4   0        dt^3/2 |
| dt^3/2   0        dt^2     0      |
| 0        dt^3/2   0        dt^2   |
```

其中：

- `sigma_a`：过程噪声强度，对应配置里的 `process_accel_sigma_mps2`

工程上怎么理解这件事：

- `sigma_a` 越大，滤波器越愿意相信“目标可能突然机动”
- `sigma_a` 越小，滤波器越坚持“目标应该沿着原趋势走”
- 它决定的是“轨迹允许有多活跃”

当前默认：

```text
process_accel_sigma_mps2 = 6.0
```

### 5.6 观测噪声

观测噪声矩阵写成：

```text
R = sigma_z^2 · I_2
```

展开后就是：

```text
R =
| sigma_z^2     0       |
|    0       sigma_z^2  |
```

其中：

- `sigma_z` 对应配置里的 `measurement_sigma_m`

它的意思是：

- 每次上报的位置都带有一个近似各向同性的位置误差
- 误差量级大约由 `measurement_sigma_m` 控制

当前默认：

```text
measurement_sigma_m = 12.0
```

### 5.7 前向 Kalman 滤波

前向滤波每一步都做两件事：先预测，再用观测修正。

预测：

```text
x_pred = F · x_prev
P_pred = F · P_prev · F^T + Q
```

这里：

- `x_pred`：预测状态
- `P_pred`：预测协方差

有了观测之后，再看“预测位置”和“观测位置”差多少：

```text
r = z - H · x_pred
```

这个 `r` 就是残差，也可以理解成“预测点离观测点还差多少米”。

为了让异常判定更直观，代码还把残差换算成了残差速度：

```text
residual_speed = ||r|| / dt_eff
```

这里的含义是：

- 如果预测点和观测点只差几米，那残差速度很小
- 如果在很短 `dt` 里突然差了很多米，那残差速度会非常大
- 这样就能更自然地拿它和“正常飞行速度量级”做比较

### 5.8 软异常与硬拒绝

当前实现有两级观测抑制。

硬拒绝：

```text
if residual_speed > hard_residual_speed_mps:
    直接跳过更新，只保留预测状态
```

默认：

```text
hard_residual_speed_mps = 35.0
```

软异常：

```text
if soft_residual_speed_mps < residual_speed <= hard_residual_speed_mps:
    R' = soft_noise_scale * R
```

默认：

```text
soft_residual_speed_mps = 25.0
soft_noise_scale = 9.0
```

这样处理的含义是：

- 软异常：降低观测权重
- 硬异常：直接不用该观测

### 5.9 RTS 后向平滑

前向 Kalman 只利用了“过去和当前”的信息。  
RTS 平滑会再从后往前走一遍，把未来点的信息补回来。

平滑增益：

```text
C_k = P_f(k) · F_(k+1)^T · inv(P_pred(k+1))
```

平滑状态更新：

```text
x_s(k) = x_f(k) + C_k · (x_s(k+1) - x_pred(k+1))
```

读法可以直接理解成：

- `x_f(k)`：第 `k` 步前向滤波后的结果
- `x_pred(k+1)`：从第 `k` 步往前推，对第 `k+1` 步做出的预测
- `x_s(k+1)`：第 `k+1` 步经过平滑后的更优结果
- 两者的差值，表示“未来信息告诉我们，前面的估计其实还可以修一点”

这一步的作用：

- 用窗口内未来信息修正过去状态
- 减少锯齿和相位滞后
- 比纯前向滤波更 smooth

### 5.10 fixed-lag 输出

实时主链路不是整段收齐后再输出，而是固定滞后。

当前默认：

```text
window_size = 5
lag_points = 2
```

含义是：

- 每次重新计算当前窗口
- 只释放距离窗口尾部 `lag_points` 之外的那个点

这就是当前“实时可用”的关键工程约束。

## 6. `robust_prefilter_kalman` 模式

实现文件：

- [prefilter.py](D:/Github/Drone-Kalman-Filter/src/drone_kalman_filter/prefilter.py)

这个模式是在 `kalman` 之前先做一层鲁棒预处理。

流程：

```text
原始局部坐标
  -> 异常判定
  -> 单点修复 或 burst 修复
  -> 视情况做中值滤波
  -> 送入 Kalman / RTS
```

### 6.1 为什么需要 prefilter

纯 Kalman 对下面这些坏样本不够强：

- 单点大跳变
- 连续 2-3 个跳点
- 来回反转的锯齿尖刺

如果直接喂给 Kalman，坏观测会污染后续状态。

### 6.2 单点异常判定

当前主要看两类条件，它们本质上都在回答一个问题：

“这个点相对前后上下文，看起来像不像正常飞出来的？”

条件一：等效速度过大

```text
speed = distance / dt_eff

如果 speed > prefilter_hard_speed_mps
就把这个点当成硬异常
```

条件二：短时间内位移过大

```text
如果 dt_eff <= prefilter_hard_distance_dt_seconds
并且 distance > prefilter_hard_distance_m
也把这个点当成硬异常
```

当前默认：

```text
prefilter_hard_speed_mps = 45.0
prefilter_hard_distance_dt_seconds = 1.5
prefilter_hard_distance_m = 80.0
```

### 6.3 bridge spike

bridge spike 主要针对“孤立尖刺”。

它的典型形态是：

- 当前点离前后点都很远
- 但前后点彼此很近

代码里会同时检查：

- 邻点距离是否足够小
- 当前点到邻点是否足够大

相关参数：

```text
prefilter_bridge_neighbor_distance_m
prefilter_bridge_center_distance_m
```

### 6.4 burst 的定义

burst 处理的是“连续几个坏点黏在一起”的情况，比如：

- 第一个点突然跳出去
- 第二个点又往反方向跳
- 第三个点再往回拐

当前把它识别成 burst，至少要满足这几件事：

```text
1. 这些点相对同一个 trusted anchor 都可疑
2. 点与点之间时间间隔很短
3. 局部方向出现明显反转
```

方向反转用余弦来判断：

```text
cos(theta) = (u · v) / (|u| · |v|)
```

当：

```text
cos(theta) < -0.5
```

就意味着夹角已经大于 120°，说明这不是“正常拐弯”，而是很像来回折返的尖刺。

### 6.5 trusted anchor

`robust_prefilter_kalman` 里维护了一个跨窗口的 `trusted_anchor`。

它的作用是：

- 作为后续 burst 修复的稳定参考点
- 避免坏点被误当成新的轨迹锚点

关键约束是：

```text
burst 内部被修复出来的点，不能推进 trusted anchor
```

只有真正看起来正常的点，才能更新 anchor。

### 6.6 burst 修复

burst 修复不是“每个坏点各修各的”，而是把整串可疑点当成一个整体处理。

思路是：

```text
左边找一个可信锚点
右边找一个未来稳定锚点
中间整段一次性补出来
```

如果未来锚点存在，就按时间做线性插值：

```text
p(t) = p_left + alpha · (p_right - p_left)

其中：
alpha = (t - t_left) / (t_right - t_left)
```

这可以理解成：

- 左右两端都可信
- 中间这几帧只是传感器乱跳
- 那么让中间点沿着“从左端平滑走到右端”的方向过渡

如果右端锚点还没出现，就先采用保守策略：

```text
整段 burst 暂时保持上一可信点
```

### 6.7 单点修复

如果只是单个孤立坏点，就没必要走整段 burst 逻辑。

当前优先级是：

```text
1. 优先插值
2. 其次外推
3. 最后才保持上一点
```

其中恒速外推写成：

```text
v = (p2 - p1) / (t2 - t1)
p3 = p2 + v · (t3 - t2)
```

读法是：

- 先用前两个可信点估一个速度
- 再把这个速度往后推一小步

### 6.8 中值滤波

中值滤波不是全局关闭，也不是全局打开。

当前策略：

```text
- legacy 单点路径：保留 median
- burst 路径：
    如果真的发生 burst 修复 -> 跳过 median
    如果只是进入 burst 检查但没有实际 burst -> 保留 median
```

目的：

- 不破坏普通去毛刺能力
- 避免 burst 修复后再被中值滤波把污染抹回正常点

## 7. baseline 的定义

实现文件：

- [tooling/drone_kalman_validation/baseline.py](D:/Github/Drone-Kalman-Filter/tooling/drone_kalman_validation/baseline.py)

### 7.1 baseline 是什么

当前项目里的 `baseline` 可以直接理解成：

```text
同一段轨迹如果不考虑实时输出，
而是等整段数据都收齐之后，
再用同一套二维恒速 Kalman + RTS 整段重算一遍，
得到的那条离线参考线
```

它不是：

- ground truth
- 上游真实轨迹
- 最终前端验收裁判

它只是：

- 离线参考线
- 技术诊断时的对照对象

### 7.2 baseline 与实时主链路的共同点

共同点：

- 同样先转局部平面
- 同样使用二维恒速模型
- 同样只观测位置
- 同样调用 [kalman.py](D:/Github/Drone-Kalman-Filter/src/drone_kalman_filter/kalman.py) 中的 `smooth_positions`

### 7.3 baseline 与实时主链路的区别

区别一：按整段处理

```text
实时主链路：fixed-lag
baseline：fixed-interval
```

区别二：不走 prefilter

```text
baseline 直接对原始观测做整段平滑
不经过 robust_prefilter_kalman 的预处理层
```

区别三：baseline 会开启 `reset_velocity_on_reject=True`

也就是：

```text
如果某一帧观测被硬拒绝：
    保留预测位置
    但把速度分量清零
    并重置速度相关协方差
```

这么做是为了尽量抑制长段漂移。

### 7.4 baseline 为什么仍然可能不可靠

即使是整段 RTS，baseline 仍可能出问题，因为它依赖：

- 分段是否合理
- 运动模型是否适配真实轨迹
- 过程噪声与观测噪声参数是否合适
- 原始观测本身是否脏到足以把整段带偏

所以 baseline 的地位是：

```text
辅助诊断 > 最终真值
```

## 8. 验收指标

实现文件：

- [metrics.py](D:/Github/Drone-Kalman-Filter/src/drone_kalman_filter/metrics.py)

### 8.1 `implied_speed_mps`

```text
implied_speed(k) =
    distance(raw_k, raw_(k-1)) / max(dt, min_dt_seconds)
```

作用：

- 识别疑似跳变
- 不是业务速度字段

### 8.2 `normal_point_offset_p95_m`

先标记跳变点：

```text
如果 implied_speed > 45 m/s
就把当前点和它后面 5 个点都排除出“正常点”集合
```

然后在剩余正常点上计算：

```text
offset = distance(raw, smoothed)
```

最后取 P95。

它回答的是：

```text
正常点上，平滑结果有没有把轨迹拉偏太多
```

### 8.3 `direction_flip_count`

对连续三点 `A, B, C`：

```text
如果：
1. |AB| > 10 m
2. |BC| > 10 m
3. angle(AB, BC) > 120°

就记一次 flip
```

它衡量的是：

```text
轨迹是否仍在高频来回折返
```

### 8.4 `recovery_points_p95`

只对很明显的大跳变统计恢复速度：

```text
如果 implied_speed > 100 m/s，
就开始计数，
直到 smoothed 相对 raw-reference 的偏移重新回到 20 m 以内
```

它衡量的是：

```text
发生大跳变后，算法恢复正常的速度
```

## 9. diagnose 图的技术含义

`diagnose` 图不是前端效果图，而是责任定位图。

它同时比较：

- `raw`
- `current`
- `baseline`

并额外画：

- `raw vs baseline`
- `current vs baseline`

它用来回答：

```text
1. current 是不是明显跑飞了
2. 如果 raw 和 current 都远离 baseline，
   那更像是 baseline 自己有问题
```

所以 diagnose 的用途是“定位责任”，不是“看起来顺不顺”。

## 10. 当前实现边界

当前实现明确不做：

- 3D 高度平滑
- 跨设备融合
- 姿态与速度字段的重建
- 复杂多源定位融合

当前工程边界很明确：

```text
先把 2D 轨迹线稳定地画顺
并且保证输入输出结构完全一致
```
